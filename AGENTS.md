# Expo HAS CHANGED

This project is on **Expo SDK 56** (upgraded from SDK 55 on 2026-07-13). Read the exact versioned docs at https://docs.expo.dev/versions/v56.0.0/ before writing any code.

When adding native modules, always use `npx expo install <pkg>` (never plain `npm install`) so versions stay pinned to SDK 56 ‚Äî mixing in mismatched-SDK packages causes native build failures (e.g. unresolved symbols like `OptimizedRecord`). If `app.json`'s `sdkVersion` field and the installed `expo` package version ever drift apart, `expo install --fix` silently resolves against the *declared* `sdkVersion`, not the installed package ‚Äî keep them in sync.

`@expo/ui` is now **stable** as of SDK 56. Its SwiftUI `TextField` is properly controlled via `useNativeState` + `text`/`onTextChange` (an `ObservableState<string>`) instead of the old `defaultValue`-only API ‚Äî see [LiquidTextField.tsx](src/components/common/LiquidTextField.tsx) for the pattern. `expo-font` and `expo-splash-screen` are now required explicit dependencies (previously implicit) ‚Äî `@expo/vector-icons` needs `expo-font` as a peer, and the top-level `splash` key in `app.json` was removed in favor of the `expo-splash-screen` config plugin. RN 0.85 also dropped `StyleSheet.absoluteFillObject` in favor of `StyleSheet.absoluteFill`.

`@gorhom/bottom-sheet` is fully removed ‚Äî `LiquidBottomSheet.tsx` already covers every sheet in the app via `@expo/ui`'s native SwiftUI/Jetpack Compose sheets, and the old `AppBottomSheet.tsx`/`BottomSheetModalProvider` were dead weight. `LiquidHeaderMenu.tsx`'s Android path was also migrated off `react-native-paper`'s JS `Menu` onto `@expo/ui/community/menu`'s `MenuView`, giving a real native Compose `DropdownMenu` there instead (iOS was already native SwiftUI `Menu` and is unchanged). `LiquidTabs.tsx` was deliberately **not** migrated to `@expo/ui/community/segmented-control` ‚Äî that component has no per-segment icon slot, and the Android tab bar (`UdhaarScreen`, others) relies on icons; the custom pill-tab implementation stays.

**Universal `@expo/ui` layer** (`import { Host, ... } from '@expo/ui'`, not `@expo/ui/swift-ui`/`@expo/ui/jetpack-compose` directly) is the SDK 56-recommended way to write one shared render path per component instead of a `Platform.OS` split. Migrated so far: `LiquidTextField.tsx` (real native Compose `BasicTextField` on Android now, not a plain RN fallback), `LiquidBottomSheet.tsx` (the universal `BottomSheet` owns its own animation-aware mount/unmount lifecycle ‚Äî don't reintroduce a manual `mounted`/`presented` state dance, it's no longer needed), and `LiquidHeaderIconButton.tsx` (uses the universal `Icon`, which needs a bundled vector asset on Android ‚Äî see `@expo/material-symbols` below).

**Icons on Android via the universal layer need `@expo/material-symbols`, not `@expo/vector-icons`.** The universal `Icon`/`Button` components require Android icons as pre-bundled XML vector-drawable assets (`require('@expo/material-symbols/<name>.xml')`), not an arbitrary RN icon component ‚Äî `@expo/vector-icons`/Ionicons can't be passed as Android icon content there. `LiquidHeaderIconButton.tsx` and `LiquidButton.tsx` each have their own `_ICON_ASSET`/`SF_TO_MATERIAL` map translating the existing SF Symbol / Ionicons-name props to the matching Material Symbols asset internally, so callers didn't need to change. `LiquidHeaderMenu.tsx`'s trigger button is the one exception left on plain Ionicons ‚Äî its Android path renders through `@expo/ui/community/menu`'s `MenuView`, whose trigger `children` slot is plain RN content wrapped in a `Pressable` (not a native-hosted slot), so Ionicons there is correct as-is, not a gap.

**`LiquidButton.tsx` is now on the universal `Button`/`Icon`/`Row`** (migrated 2026-07-13, deliberately after `LiquidBottomSheet`/`LiquidHeaderIconButton` proved out the pattern, given this is the single most-used component in the app ‚Äî 27 call sites). Two non-obvious things if you touch this file:
- Universal `Button`'s `'filled'` variant maps to SwiftUI's `borderedProminent`, not Liquid Glass ‚Äî the real glass look only comes back because a `buttonStyle('glassProminent')` modifier is passed explicitly through the `modifiers` escape hatch, which overrides the variant-derived style (confirmed via `@expo/ui`'s own `omitUserOverridden`).
- **Never pass the `disabled` prop to universal `Button`.** It compiles fine but silently applies SwiftUI's `.disabled()` modifier on iOS, which forces a flat neutral-grey override that defeats the whole point of tinting a disabled button ‚Äî this was already fought and documented once before the universal-component migration (see git history / the original PressableScale-era comment) and got silently reintroduced once already while writing this migration. Disable presses only via `onPress={isDisabled ? undefined : onPress}` plus `opacity()` dimming in `modifiers`.

`ConfirmDialogProvider.tsx`'s `Text` is on the universal layer (pure pass-through wrapper on iOS, zero functional gap); its `Host`/`Alert`/`Button` stay on `@expo/ui/swift-ui` ‚Äî neither `Alert` nor `ConfirmationDialog` exist in the universal export list at all, and the `Button`s need the swift-ui-only `role` prop (`'destructive'`/`'cancel'`) for correct native styling, which the universal `Button` doesn't expose (not even via the `modifiers` escape hatch ‚Äî checked, no `role` modifier exists).

**Uses `Alert`, not `ConfirmationDialog`** (switched 2026-07-13). SwiftUI's `confirmationDialog` presents as an action sheet ‚Äî no real enter animation, abrupt appear, minimal fade on dismiss ‚Äî which is wrong for how this app actually uses it: always as a single full-screen modal-style prompt, never a real "pick one of several options from a sheet" action sheet. `Alert` is SwiftUI's actual centered modal dialog (real native scale/fade transitions) and has the identical `Trigger`/`Actions`/`Message` shape, so this was a straight swap, not a rewrite. If a future screen genuinely needs action-sheet semantics (many peer options, not a yes/no/destructive prompt), that's the one case `ConfirmationDialog` would actually be the righter primitive ‚Äî don't blanket-revert this.

`LiquidHeaderMenu.tsx`'s `Host` and both button layers (trigger icon + individual menu-item buttons) are now on the universal layer too ‚Äî its `Button` usage doesn't need `role`, unlike `ConfirmDialogProvider`'s. Only `Menu`/`Section` (SwiftUI's own menu composition, no universal equivalent) stay on `@expo/ui/swift-ui`. One gotcha hit while doing this: universal `Button` defaults to `variant="filled"` (SwiftUI `borderedProminent`) ‚Äî inside a `Menu`, that would paint each row as a bordered pill instead of a plain menu row, since the original bare `swift-ui` `Button` had no `buttonStyle` at all. Pass `variant="text"` (‚Üí `plain`) explicitly for any `Button` meant to render as a plain row/menu item.

`LiquidTabs.tsx` is the last file still fully on `@expo/ui/swift-ui` ‚Äî universal `Picker` (used there) has no segmented-control appearance, only `'menu'`/`'wheel'`. Nothing left to migrate there. Before "fixing" any of these later, re-check `node_modules/@expo/ui/build/universal/index.d.ts` first ‚Äî these decisions were made by reading that file directly, not from memory.

# Performance conventions (2026-07-13)

Two zustand antipatterns were swept app-wide ‚Äî both are now the established convention, don't reintroduce either:

1. **Never subscribe to a whole store with a bare `useAppStore()` / `useOnlineShopStore()`** (no selector). Every screen used to do this, meaning any store field changing anywhere in the app re-rendered every screen holding a reference to it, regardless of whether that screen used the changed field. Every call site now uses `useShallow(state => ({ field: state.field, ... }))` ‚Äî same destructured shape, just wrapped. If you add a new field to a screen's store usage, add it inside the `useShallow` selector object, not as a second bare `useAppStore()` call.
2. **`FlatList` row components must be `React.memo`'d, with stable callback props.** Passing a fresh inline arrow function as a prop (`onPress={() => doThing(item)}`) defeats `React.memo` even if the row component itself is memoized ‚Äî a "new" function reference every render always fails the shallow-equality check. The established pattern (see `ProductCard.tsx`/`InventoryScreen.tsx`, or `BillRow`/`BillHistoryScreen.tsx` for a fuller example): the row component takes an item-aware callback (`onPress: (item: T) => void`), the *screen* passes a single stable top-level handler (`useCallback`, or already-stable from elsewhere) directly ‚Äî never wrapped ‚Äî and the row itself does `onPress={() => onPress(item)}` internally (that inner closure is fine; it only affects whether the inner `TouchableOpacity` re-renders, not whether the outer memoized row does).
   - Any translation string (`t('key')`) needed inside a memoized row must be precomputed once in the parent and passed down as a plain string prop, never the `t` function itself (not stable across renders).
   - `const s = makeStyles(colors)` must be wrapped in `useMemo(() => makeStyles(colors), [colors])` in any screen with a memoized row ‚Äî otherwise `StyleSheet.create` returns a new object every render regardless of whether `colors` changed, which alone defeats any child memoization relying on `s` as a prop.
   - Per-row `O(n)`-scan lookups (e.g. `returns.filter(r => r.billId === billId)` called inside `renderItem`) should be precomputed once into a `Map` via `useMemo` keyed by id, then looked up per row in O(1) ‚Äî see `returnsByBill` in `BillHistoryScreen.tsx` or `supplierStats` in `SupplierScreen.tsx`.

**Product photos use `expo-image`, not RN's built-in `Image`.** Plain RN `Image` has weak/inconsistent disk caching (doesn't persist across app restarts on iOS) ‚Äî `expo-image` gives real disk+memory caching, so a product photo scrolled past in a `FlatList` doesn't refetch from network when it recycles back into view. `import { Image } from 'expo-image'` is a drop-in replacement for `source={{ uri }}` usage (same default `contentFit: 'cover'` as RN's default `resizeMode: 'cover'`) ‚Äî always use it for any new network-image usage, never RN's `Image`.
